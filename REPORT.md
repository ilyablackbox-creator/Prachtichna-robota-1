**1. Опис проекту**

У межах роботи було створено програмне забезпечення для цифрового обліку навчальної групи. Основний функціонал включає реєстрацію студентів, моніторинг академічних успіхів через журнал оцінок, автоматизацію розрахунку рейтингів та експорт сформованих списків у структурований формат JSON для подальшого використання.



**2. Технічна архітектура**

Програмна логіка розподілена між трьома ключовими модулями:



Student: Базовий об'єктний тип, що описує студента. Реалізовано суворий контроль доступу до даних через механізм властивостей, а також валідацію вхідної інформації (ПІБ, номер квитка) на етапі ініціалізації.



GradeJournal: Спеціалізований контейнер для зберігання та обробки балів, що використовує словникову структуру Dictionary<string, double> для швидкого доступу до предметів.



StudentGroup: Адміністративний модуль, що відповідає за цілісність колекції об'єктів, реалізує алгоритми пошуку, сортування та забезпечує взаємодію з файловою системою.



**3. Робота з Git**

Весь процес розробки було зафіксовано в системі контролю версій для забезпечення цілісності коду:



Ініціалізація: Створено локальне сховище (git init) та налаштовано .gitignore для ігнорування службових папок bin/ та obj/.



Гілки: Розробка велася в окремих гілках (feature-branches), що дозволило тестувати різні функції (моделі, валідацію, серіалізацію) ізольовано від основної гілки main.



Коміти: Кожна логічна частина роботи (наприклад, додавання Regex-валідації) фіксувалася окремим комітом з описом змін.



Синхронізація: Виконано прив'язку локального проекту до GitHub та завантаження коду командою git push -u origin main.







**4. Питання:**



**2.1.** Що таке інкапсуляція? Наведіть 5 різних прикладів з вашого коду.



Інкапсуляція — це механізм ООП, який об'єднує дані (поля) та методи для роботи з ними в одному об'єкті, обмежуючи прямий доступ до внутрішнього стану об'єкта ззовні. Це дозволяє контролювати коректність даних.

Приклади з коду:

1\)	Приватні поля: Поле private string recordBookNumber приховане від прямої зміни, доступ можливий лише через властивість.

2\)	Властивість з валідацією: Властивість FullName перевіряє довжину рядка перед записом у приватне поле.

3\)	Приватний setter: У властивості AverageGrade встановлено private set, тому змінити бал можна лише через спеціальний метод UpdateAverageGrade.

4\)	Приватний список у групі: Поле private List<Student> students у класі StudentGroup недоступне для прямої модифікації зовні.

5\)	Методи класу: Метод CalculateAge() приховує складну логіку обчислення віку, надаючи лише готовий результат.



**2.2.**	Порівняйте авто-властивості та властивості з повною реалізацією. У яких випадках ви обирали кожен варіант?

Авто-властивості (Status, Notes): Використовувалися для даних, які не потребують додаткової логіки при читанні або записі. Це робить код лаконічнішим.



Повна реалізація (FullName, RecordBookNumber): Обиралася, коли необхідна валідація (перевірка довжини, формат Regex).



Вибір: Якщо є бізнес-правила (наприклад, "тільки 4 цифри"), використовувалася повна реалізація.





**2.3.**	Для яких даних ви застосували валідацію? Які види валідації використовували?



Валідація була застосована для полів:



1\)	FullName: Перевірка на null та мінімальну довжину (5 символів).



2\)	RecordBookNumber: Використання Regex (@"^\\d{4}$") для забезпечення формату рівно з 4 цифр.



3\)	PersonalEmail: Перевірка наявності символу @ та . для базової перевірки формату пошти.



4\)	AverageGrade: Обмеження діапазону від 0 до 100 балів.





**2.4.** 	Навіщо потрібні read-only та init-only властивості? Приклади з коду.

* &#x09;Read-only (get): Властивість Age. Вона не має поля, а лише обчислює значення на основі дати народження. Це гарантує, що вік завжди актуальний і його не можна змінити вручну.
* &#x09;Init-only (init): Властивості DateOfBirth та EnrollmentDate. Вони дозволяють встановити значення лише під час створення об'єкта, після чого ці дані стають незмінними (immutable).



**2.5.** 	Поясніть роботу властивості Age. Чому вона обчислювана?

Властивість Age є обчислюваною, тому що вік людини постійно змінюється з плином часу. Зберігати його як статичне число в базі недоцільно. Вона реалізована через метод CalculateAge(), який порівнює поточну дату з DateOfBirth, що забезпечує точність до дня.



**2.6.** 	Як ви реалізували принцип єдиної відповідальності в класах?

Кожен клас у моєму проєкті відповідає за одну частину системи:

* &#x09;Student: Тільки дані про конкретного студента та його особисту поведінку (вік, валідація ПІБ).
* &#x09;GradeJournal: Тільки збереження оцінок та розрахунок середнього балу по предметах.
* &#x20;       StudentGroup: Тільки логіка управління колекцією студентів (пошук, видалення, статистика всієї групи).
